# 面向无姿态先验室内图像的轻量化多特征三维重建

版本：2026-07-16 第一版正文

## 摘要

室内场景中存在大面积白墙、玻璃、重复纹理和遮挡，仅依靠局部图像匹配容易产生相机位姿漂移，也难以直接得到可编辑的结构表示。本项目研究一组无序或弱位姿先验的室内图像，目标不是生成更密的点云，而是在统一三维坐标中恢复带有可见支撑范围的平面实例，并补充能够反映墙角、门窗边缘等结构的三维线段。系统以 DUSt3R 的点图预测和全局对齐为公共几何底座，在二维阶段预测平面 support，经局部几何检验后，使用显式的 `(alignment_view_index, x, y)` 键映射到对齐点图。最终方法没有直接采用学习分组作为平面结果，而是用其生成并排序平面假设，再在完整点云上执行共识检验、置信度加权重拟合和连通域分割。这样保留了 RANSAC 对异常点的鲁棒性，同时利用了图像分支提供的实例线索。

实验固定使用相同的 DUSt3R 全局缓存，在 8 个独立 Structured3D 场景上比较全局 RANSAC、直接拟合、人工 support 合并和学习 support 引导 RANSAC。最终方法的平均配对 F1 从 `0.704999` 提高到 `0.744248`，8 个场景均获得正增益；平均 matched IoU 从 `0.614389` 提高到 `0.685150`，平均 overmerge excess 从 `2.500` 降至 `1.125`。[E2] Stage1 在 80 个固定配对记录上的平均 precision、recall 和 F1 分别为 `0.883814`、`0.629410` 和 `0.724166`。[E1] 五视图 DUSt3R 推理与 300 步全局对齐的中位耗时为 `9.123 s`。实验没有支持普遍加速结论，因此本项目只保留质量改进主张。结果表明，在训练资源有限的条件下，把学习 support 作为几何估计的引导信息，比直接用预测标签替代全局几何分组更稳妥。

关键词：无姿态先验三维重建；室内平面；DUSt3R；RANSAC；结构线；可编辑几何

## 1 研究背景与任务定义

多视图三维重建通常依赖可靠的特征匹配和相机位姿。在纹理丰富、视角变化有限的图像中，这条路线已经较成熟，但室内环境往往不满足这些条件。白墙和天花板缺少局部纹理，玻璃会产生反射，门窗和柜体又包含大量相似边缘。即使点云在视觉上较完整，属于同一面墙的点也可能因位姿误差形成数个近似平行的薄层。若随后只根据法向和偏移量合并平面，很容易把相邻结构合并，或者把同一结构切成多个实例。

DUSt3R 一类点图模型减少了对传统 SfM 初始化的依赖[1]。它可以从图像对预测三维点和置信度，再通过全局优化把多个视图放入共同坐标系。不过，点图本身仍是稠密采样。它没有直接回答三个问题：哪些点构成同一平面实例；一个平面的可见边界在哪里；二维预测怎样无歧义地对应到全局对齐后的三维点。项目早期曾尝试只调节 pair-local 平面合并阈值，也尝试将平面残差反馈到全局对齐。前者混淆了不同图像对的局部坐标，后者虽然降低了优化目标中的平面项，却没有改善注册后的结构指标。这些负结果促使系统回到更可检验的任务：固定 DUSt3R 几何，用明确的像素 provenance 连接二维 support 与全局点图，再评价平面身份、支撑范围和几何误差。

本文中的平面 primitive 不只是法向量和偏移量。每个输出还需要保存三维支撑点、来源视图、像素坐标和有界连通分量。这样的结果可以单独着色、选择和修改，也能与结构线一起导出为 NPZ、PLY 或 GLB。项目的完成标准是得到可复现的多场景流程和受控对比，而不是在不兼容的公开协议之间宣称领先。

## 2 相关工作与本项目边界

现有研究已经覆盖单图像平面分割、稀疏视图平面对应、已知位姿下的多视图平面融合，以及从点云中进行平面配置优化。PlaneRCNN、PlaneAE 和 PlaneTR 说明了掩码、关联嵌入和查询式平面预测的有效性[3-5]。SparsePlanes、PlaneFormers 与 NOPE-SAC 进一步处理稀疏视图中的平面对应和相对位姿[6,14-15]。PlanarRecon、AirPlanes 等工作研究了多视图跟踪、融合和几何基线[8,10]。DUSt3R、MASt3R 及后续平面相关模型则把研究重点推进到弱位姿先验的点图重建[1-2,18]。

传统多视图重建通常先估计相机，再通过三角化、深度融合或多视图立体得到点云。其优点是各模块含义明确，缺点是前端匹配失败会传递到后续步骤。DUSt3R 将图像对直接映射为带置信度的点图，并用全局对齐统一多视图[1]。MASt3R 在匹配能力上继续扩展这一路线[2]。它们为弱纹理和未知位姿场景提供了更强的公共几何，但输出仍以点为主。Plane-DUSt3R 等工作已经研究点图模型与房间平面结构的结合[18]，因此本文不能把 pointmap 加平面先验本身写成创新。

单图像平面方法更关注实例掩码、平面参数和深度。PlaneRCNN 同时预测平面区域和几何[3]，PlaneAE 使用关联表示处理数量不固定的平面实例[4]，PlaneTR 采用查询并引入结构线线索[5]。这些工作证明二维 appearance 对平面身份有帮助，但单幅图像无法独立解决跨视图尺度和全局坐标一致性。SparsePlanes 与 PlaneFormers 把平面对应和相对位姿放到稀疏视图设置中[6,14]，NOPE-SAC 则使用由平面产生的位姿假设[15]。这些方法与本项目最接近的部分是以学习信号缩小几何假设空间。

在已知或较可靠位姿下，PlaneMVS、PlanarRecon 和 AirPlanes 研究了平面检测、跟踪与融合[7-8,10]。AirPlanes 的结果尤其说明，简单而强的几何 RANSAC 仍然是必须认真比较的基线。点云侧的 PolyFit 和 Good Configurations of Planar Primitives 进一步表明，从候选平面构造有界面片、优化平面数量与点分配也已有成熟研究[9,11]。本项目没有尝试生成封闭多边形房间模型，而是保留观测到的连通支撑点。这样做牺牲了模型完整性，但减少了对不可见边界的猜测。

线特征方面，经典 LSD 可以在不训练额外网络的情况下给出亚像素级二维线段[26]。三维线重建通常还需要跨视图匹配和遮挡处理。本项目只在单个对齐 pointmap 内提升线段，并用深度间断切分，因而更接近结构提示而不是完整的多视图线地图。把它加入系统的原因很实际：平面内部纹理较弱时，墙角和门窗边缘仍能帮助观察重建是否发生错位；但本次实验没有用线段反向优化相机或平面。

因此，本项目不把平面查询、SVD 拟合、RANSAC 或二维线检测单独称为新算法。可被实验支持的方法改动较窄。第一，系统使用明确的视图索引和点图像素坐标保存 support provenance，禁止通过数组形状猜测二维到三维的对应关系。第二，学习分组只负责提出局部平面假设和提供假设优先级，最终内点来自完整全局点云上的距离共识，参数由 DUSt3R 置信度加权重拟合。结构线、批处理审计和可编辑导出属于系统实现，不单列为算法创新。

## 3 系统总体结构

输入是同一室内空间的若干 RGB 图像，图像之间不要求给定相机位姿。系统首先建立完整图像对，使用固定 DUSt3R 权重预测 pair-wise 点图和置信度，再运行全局对齐，得到每个视图在统一坐标中的 `scene.get_pts3d()`。Structured3D 原图为 `1280 x 720`，本次五视图计时中按 DUSt3R 的 image-size-512 设置载入为 `512 x 288`。后续模块只能通过已登记的 resize、crop 和视图顺序访问点图，不能把 Stage1 掩码坐标直接当作全局点图坐标。

平面支路分两级。Stage1 从图像和 DUSt3R 特征预测实例 support。Stage2 对候选区域计算局部点云几何特征，判断区域是否需要合并，并进行局部平面重拟合。每条 support 记录保存 `alignment_view_index` 与 `(x, y)`。Stage3 使用这三个整数组成精确连接键，将记录映射到全局点云缓存。重复观测、冲突标签和无法匹配的键分别计数，不在后台静默删除。

结构线支路使用 OpenCV LSD 检测二维线段。系统沿线段采样同一张对齐点图，根据置信度和深度跳变切分，再对有效三维样本做鲁棒线段拟合。该支路平均每个场景耗时 `1.725 s`。[E1] 从可视化结果看，门窗框和墙角能形成较稳定的线段，窗内小物体和强反射区域仍会产生短线。考虑到这些线尚未进入最终平面优化，本文把它们作为独立结构输出和定性补充，不声称它们已经提高了平面重建指标。

## 4 学习 support 引导的平面 RANSAC

### 4.1 精确 support 映射

对全局缓存中的每个点，系统记录其视图编号和点图像素坐标。Stage1、Stage2 输出使用相同约定。映射时先把 `(view, x, y)` 打包成整数键，再执行精确连接。一个键若对应多个互相冲突的预测标签，会被记录为 conflict。实验同时保留 raw support 和 conflict-drop 两种统计，但后者只用于消融。原因是丢弃冲突会大幅降低覆盖率，某些配置能得到看似更高的 F1，却只评价了极少量容易点。最终方法不以这种分数作为主结果。

### 4.2 局部引导假设

对每个学习 support 标签，系统收集已经映射到全局缓存的唯一点。点数不足的候选直接拒绝。其余候选先做置信度加权 SVD，再进行固定次数的三点采样，以距离阈值内的点数和中位残差选择局部平面。局部内点数和内点比例均达到阈值后，该平面才进入全局候选集合。这一步没有把 support 外的点排除在外，它只是把全局随机搜索缩小为一组由图像分组提出的平面方向和位置。

### 4.3 全局共识、重拟合与有界分量

顺序提取阶段在尚未分配的完整点云上评价每个候选。候选分数由估计的全局内点数与仍然有效的学习 support 内点数相加得到。分数最高的候选先被选择，然后在完整点云上重新计算距离内点，并使用 DUSt3R 置信度做加权平面拟合。拟合后的无限平面内点经过欧氏或体素连通分量分割，形成有界支撑。为避免同一主平面被其他局部候选重复发现，系统按照基线相同的规则移除该无限平面的全部内点。学习分组未覆盖的剩余点可进入一次缩减的随机 RANSAC fallback。

这种设计保留了两类信息。图像分支擅长区分在几何上相近但外观和位置不同的区域，完整点云共识则能补回 Stage1 漏掉的同平面点。Stage1 的平均 precision 为 `0.883814`，recall 只有 `0.629410`。[E1] 这个差距说明直接把 support 标签当作最终分割会损失大块平面，而将其用于假设生成更符合模型的实际能力。

### 4.4 输出与可追溯信息

每个最终平面保存法向、偏移、全局点索引、连通分量和来源 support 诊断。点云结果可导出为 PLY，平面参数同时写入 JSON、TXT 和 NPZ。NPZ 使用版本字段区分旧的 pair-local 文件与新的全局缓存，避免下游脚本把两类坐标混用。结构线文件保存二维端点、三维端点、所属视图、采样数和拒绝原因。对人工检查而言，overlay 图比单独的数字更容易暴露问题，例如线段是否跨过深度断层、窗框是否被重复提升、同一面墙是否出现多个近似平行颜色块。

全局缓存只生成一次，各候选方法以只读方式复用。缓存中的 view registry 保存原图路径、对齐视图索引、点图尺寸和坐标约定。批处理 manifest 再记录缓存 SHA256 与各预测文件 SHA256。这样做使方法比较与 DUSt3R 的偶然对齐波动分离，也使失败场景能够回到具体的视图和像素检查。

## 5 数据、指标与实验协议

实验使用 Structured3D[28] 的 8 个独立场景，每个场景保留一个五视图房间或 perspective 组。场景选择、Stage1/Stage2 checkpoint、DUSt3R 权重、图像顺序和所有阈值在最终批处理前冻结。所有方法读取相同的 DUSt3R 全局缓存，避免把重复对齐造成的随机变化误认为方法增益。批处理记录 Git SHA、输入路径、SHA256、硬件、运行时间和失败阶段，并拒绝覆盖已有输出。[E3]

平面评价同时关注实例身份、支撑和几何。身份指标包括 plane precision、plane recall、配对 F1、fragmentation excess 和 overmerge excess。支撑指标包括 matched IoU、GT coverage 与 assignment rate。几何指标包括公共有效匹配对上的法向角误差和点到平面残差。由于某些方法可能通过只保留少量简单点获得较高准确率，本文要求 F1 与 coverage 同时报告。最终方法的提升门槛在查看八场景结果之前确定，包含场景胜率、配对增益、覆盖率和过合并约束。

效率测试只做推理，不训练或调参。Stage1 准确率来自最终 8 场景对应的 80 个固定 pair 记录。GPU 部分分别测试 DUSt3R 两图像 backbone、单图像 Stage1 head、64 个候选对的 Stage2 MLP，以及五视图推理加全局对齐。每项先预热，再报告 P50、P95 和显存峰值。下游 RANSAC、结构线和评价耗时读取最终批处理保存的 runtime，不重新执行。

为减少选择性汇报，三场景 smoke 和八场景 final 使用不同 gate。Smoke 只用于验证代码、指标方向和阈值是否可执行。最终场景确定后不再根据单场景结果修改阈值。配对 bootstrap 和 sign test 由最终审计脚本一次性生成，属于不确定性描述，不参与事后选择方法。最终 decision、每场景表和输入哈希同时保存。如果主方法没有通过质量路径，报告将回退到全局 RANSAC，并把引导方法作为普通消融。

## 6 实验结果与分析

### 6.1 Stage1 support 与模型开销

Stage1 的平均 pairwise precision、recall 和 F1 为 `0.883814`、`0.629410` 和 `0.724166`，平均 predicted purity 为 `0.928410`，GT completeness 为 `0.726850`。assignment rate 和 GT-labeled coverage 均为 `0.998100`。[E1] 高 purity 与较低 completeness 同时出现，说明模型更容易找到平面内部的稳定区域，边界、遮挡和弱纹理位置仍有遗漏。

共享 DUSt3R backbone 包含 `571,171,208` 个参数，checkpoint 为 `2179.151 MiB`。Stage1 head 包含 `79,906,572` 个参数，checkpoint 为 `304.898 MiB`，约为 backbone 参数量的 14.0%。Stage2 MLP 只有 `203,521` 个参数和 `0.783 MiB` checkpoint。[E1] 因此，报告不把 Stage1 称为很小的预测头。轻量化主要体现在复用同一个点图 backbone、没有再训练第二个三维基础模型，以及后端采用小型 MLP 和确定性几何算法。

在固定环境中，DUSt3R 两图像 backbone 的 P50/P95 为 `148.428/152.940 ms`；Stage1 单图像 head 为 `75.160/101.401 ms`，增量峰值显存为 `361.899 MiB`；Stage2 一次处理 64 个候选对仅需 `0.451/0.483 ms`。五视图推理和 300 步全局对齐的 P50/P95 为 `9.123/11.922 s`。[E1] 本机没有编译 CUDA 版 RoPE2D，所以这些数据用于说明项目实际成本，不用于推断其他硬件上的极限速度。

### 6.2 最终平面方法

全局 RANSAC 在 8 场景上的平均配对 F1 为 `0.704999`，学习 support 引导方法为 `0.744248`，平均和中位配对增益分别为 `+0.039248` 和 `+0.035580`。8 个场景全部获得正增益。场景配对 bootstrap 95% 区间为 `[+0.015288,+0.063010]`，精确双侧 sign test 的 `p=0.007812`。这些统计只描述当前冻结协议，不等同于公开 benchmark 上的通用显著性结论。[E2]

支撑质量也发生了变化。平均 matched IoU 从 `0.614389` 提高到 `0.685150`，平均 overmerge excess 从 `2.500` 降到 `1.125`。7 个双方均存在有效法向匹配的场景中，引导方法的平均法向误差为 `2.195663` 度，基线为 `2.974294` 度。[E2] 一个基线场景没有有限法向值，因此该角度结果不能与包含 8 场景的其他均值混写。

最终 gate 通过质量路径，没有通过效率路径。归档 stage runtime 中，global RANSAC 的平均和中位耗时为 `20.946 s`、`15.905 s`，引导方法为 `19.172 s`、`16.631 s`。[E1] 两种聚合给出的快慢关系不同，说明场景结构和异常长尾会影响结果。本文不声称引导方法普遍更快，它的价值是减少错误合并并提高实例 F1。

### 6.3 消融与负结果

直接使用 raw manual support 身份进行融合只在 8 个场景中的 3 个胜过 RANSAC，中位 F1 差为 `-0.106110`。[E2] 这表明二维标签经过全局映射后仍包含重复观测、冲突和漏检，不能直接取代几何共识。Conflict-drop 在某些局部指标上更高，但 manual 和 direct 路径的中位 full-cache coverage 分别只有 `0.014547` 和 `0.000037`。[E2] 这些结果只作为覆盖率崩塌的诊断，不作为主方法。

平面反馈实验是另一项负结果。加入平面项后，优化日志中的 plane loss 持续下降，但注册点云上的结构指标没有同步改善。可能原因是错误的早期平面关联把局部噪声固化为约束，或者平面残差鼓励点落到某个无限平面，却没有保证实例边界和跨视图身份正确。项目因此冻结该路线，只在单场景消融中报告，不扩展到最终 8 场景。

### 6.4 定性结果与典型失败

结构线 overlay 中，长墙角、窗框外边缘和窗台通常能在相邻视图中重复出现。LSD 也会检测到抱枕条纹、书本边缘和高光边界。三维提升利用点图深度连续性过滤了一部分错误，但没有语义分类，因而不能保证每条线都属于建筑结构。展示图应同时保留成功视图和杂乱视图，不能只裁剪最整齐的门窗。

平面结果主要有三类失败。第一类是 coplanar disconnected surface。几何上共面的两块区域可能属于不同物体，若连通半径过大就会错误合并。第二类是 weak support fragmentation。Stage1 在边缘和遮挡处 recall 较低，同一墙面可能先产生多个候选。全局共识能补回部分内点，但若 DUSt3R 点图本身出现双层，候选仍可能被切开。第三类是 dominant plane suppression。顺序 RANSAC 接受一个平面后会移除其全部无限平面内点，这能避免重复检测，也可能误删邻近的薄结构。最终结果降低了平均 overmerge，但没有消除这些现象。

从八场景结果看，引导方法的增益不是由覆盖率下降换来的。其 GT coverage 中位数保持在 `0.999828`。[E2] 与之相反，conflict-drop 的分数提升伴随接近零的 full-cache coverage。这个对比说明评价平面实例时不能只看 precision 或 F1。每个方法都必须同时给出被分配的全局点比例和 GT 覆盖范围。

## 7 工程实现与复现

项目代码使用 Python、PyTorch、NumPy、SciPy 和 OpenCV。DUSt3R 作为 vendored 依赖保存在仓库中，服务器入口脚本固定使用 `lightrecon` Conda 环境，并检查 `roma`、OpenCV、NumPy、SciPy、PyTorch 与 CUDA。早期运行曾因系统 Python 中没有 torch、`roma.__version__` 不存在以及 `trimesh` 缺失而中断。后续脚本改为显式指定 Python，通过 `importlib.metadata` 查询包版本，并在需要导出 GLB 时检查 trimesh。环境错误在创建输出目录之前报告，避免把半次运行误认为完整实验。

最终实验按 selection、materialization、preflight、execution、audit 和 efficiency 六个步骤组织。Selection 从候选列表选择独立 scene ID，禁止把同一 scene 的多个 perspective 当作独立样本。Materialization 只补齐缺失的 Stage1/Stage2 文件，已经存在的结果保持只读。Preflight 校验每组视图数量、记录数和路径。Execution 对每个方法保存 stage runtime 与失败位置，即使单个场景失败也保留一行。Audit 在 CPU 上读取归档结果，重新执行冻结 gate，但不重算 DUSt3R 或 RANSAC。Efficiency 最后加载固定 checkpoint 做计时，不修改权重。

服务器网络并不稳定，几次 `git pull` 因 HTTP2 framing、443 超时或 SSL timeout 失败。实验记录因此始终打印实际 Git SHA，而不是假设 pull 成功。W3 最终运行的 SHA 为 `b5d80c2`，它包含缓存图像路径读取修复。后续 `123e9d5` 只增加环境诊断，对实验计算没有影响。[E1] 这种版本区分看似琐碎，却能避免把未拉取的新提交错误写进实验说明。

系统不会覆盖已有输出目录。每次重试使用新的版本后缀，失败日志继续保留。例如 W3 的 `_v1` 记录了缓存 schema 读取错误，`_v2` 才是有效结果。最终报告中的数字只从 `_v2/efficiency_results.json`、最终 audit JSON 和 aggregate CSV 读取。终端截图可以解释运行过程，但不作为修改数字的依据。

## 8 局限性

第一，最终实验只有 8 个 Structured3D 场景，足以完成预注册的课程项目比较，但不能覆盖真实室内图像中的运动模糊、曝光变化和复杂反射。第二，系统依赖 DUSt3R 对齐质量。引导 RANSAC 可以减少分组错误，却不能修复严重的全局位姿或尺度漂移。第三，Stage1 recall 仍然偏低；fallback 能补充遗漏几何，但也会重新引入随机 RANSAC 的过合并风险。第四，结构线目前是独立输出，没有参与平面参数优化，因此只能说明系统具备点、线、面联合表达，不能据此声称线特征提高了重建精度。

工程上还有两个限制。Stage1 head 的 checkpoint 接近 305 MiB，不适合称为极小模型。当前 RoPE2D 使用慢速 PyTorch 实现，效率数字包含这部分开销。后续若继续研究，应先在更多真实场景上复核 provenance 映射和 guided consensus，再考虑压缩 Stage1 或把可靠的线和平面邻接关系加入优化。课程截止前不再启动新的基础模型训练。

## 9 结论

本项目完成了一个无给定位姿的室内结构化重建流程。系统把二维平面 support 通过显式视图和像素键映射到 DUSt3R 全局点图，输出有界平面和三维结构线。最终采用学习 support 引导的顺序 RANSAC，而不是直接使用学习标签或修改全局对齐。在 8 个冻结场景中，该方法相对全局 RANSAC 将平均配对 F1 提高 `0.039248`，并减少了过合并；效率实验没有支持加速主张。项目同时保留了 manual identity、conflict-drop 和平面反馈的失败结果。对本次科研实践而言，主要收获不是提出一个替代 DUSt3R 的大模型，而是建立了从坐标约定、全局映射、受控比较到结果审计的完整实验链条。
